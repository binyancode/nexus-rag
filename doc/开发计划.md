# 法规检索系统 · 开发计划

> 基于《法规检索系统设计.md》（两层图 + SQG 编译 → 优化器 → 执行器）。
> 现状：`backend/` 已具备通用框架骨架（IoC 容器 / 配置系统 / API 装饰器 / MSAL 认证 /
> SQL Server / Key Vault 凭据 / API 日志）；`frontend/`（Vue + .NET BFF）为登录 + 凭据管理骨架；
> 检索引擎（`app/nexus/`）尚待自建。
>
> **开发分两大阶段：**
> - **第一部分 · 建立索引（离线）**：定数据模型 → 建三存储 → 跑索引管线 → 用测试文档验证并在图上可视化浏览。
> - **第二部分 · 查询索引（在线）**：物理算子 → SQG 编译 → 优化器 / 执行器 → 查询 API + 可视化 → 评估。

---

# 第一部分 · 建立索引阶段（离线构建）

> 目标：把法规文本变成「实体层 + 块层」两层图并落三个存储，产出**可被查询、可被浏览**的索引。

## A0 · 索引侧决策点（对应设计 §5）

动手前先冻结与索引相关的决策，避免数据模型反复返工。

- **切块粒度**：条款级 vs 整条法规（影响 `fullname` 层级与引用边密度）
- **实体类型清单**：一期建议只建 `AppType / Reg / Category`
- **边类型子集**：一期先做 `requires / belongs_to / evidence`；`supersedes / references` 二期

**产出**：`docs/decisions.md`（索引部分），冻结上述选择作为后续 SOT（单一事实来源）。

---

## A1 · 数据模型与存储基座（对应 §1、§4.0）

1. Pydantic 模型（放 `app/nexus/models/`）：
   - `Block(fullname, text, vector, summary?, keywords?)`
   - `Entity(id, type, name, aliases[], status)`
   - `Edge(src, type, dst, weight, evidence)`
2. 三个存储接口 + 实现：
   - **块向量库** → Azure AI Search 索引（向量字段 + `type` / `fullname` 过滤字段）
   - **实体索引** → AI Search 关键词 / 别名索引（或复用 SQL）
   - **边表 / 图存储** → SQL Server `nexus.edge(src, type, dst, weight, evidence)`
3. 建表脚本；`bootstrap.register_services` 追加注册三个 store。

**里程碑**：能手工写入并读回一条实体、一条边、一个块。

---

## A2 · 离线索引管线（对应 §2）

```
原始法规文本
  → ① 语义切块，赋 fullname          （结构树天然成型）
  → ② 抽实体并归一 → 实体层节点
  → ③ LLM 建边：候选提名 + LLM 裁定
  → ④ 向量化每个块
  → ⑤ 落库：AI Search（块向量+字段）+ 边表 / 图存储
```

- 切块器：按语义切分并赋 `fullname` 层级地址
- 实体抽取 + 归一（GCP 多写法 → `Reg:GCP`）
- **LLM 建边**：便宜手段（相似度 / 正则）只负责提名候选对；类型 / 方向 / 权重全交 LLM 裁定
  （不在建边环节加硬编码规则）
- 批量 embedding 并写入三个 store
- 用 `nexus/core/run_log` 记录建库进度

**里程碑**：喂一小批真实法规，产出可查询的两层图。

---

## A3 · 测试与可视化浏览（索引阶段重要测试内容）

> 目标：先打通「数据 → 索引 → 看得见」的最短闭环，用真实文档验证 A1/A2 的数据模型与索引管线，
> 并提供一个可交互的图浏览界面。

### 任务 1 · 从 `search.txt` 挑测试文档

- 从 `search.txt` 里挑一小批**能体现当前设计**的法规文档做测试集，优先满足：
  - 覆盖**多种实体类型**（`AppType` 申报类型 / `Reg` 法规 / `Category` 分类）；
  - 能产生**多种边**：`requires`（IND/NDA/IIT → 法规）、`belongs_to`（法规 → 细胞治疗等分类）、`evidence`（概念 ↔ 原文块）；
  - 能复现设计中的两个端到端示例（差集 / 交集），即存在「IND 要而 NDA 不要」「细胞治疗 ∩ IND」这类可对比的集合关系。
- 产出：`doc/test-corpus/`（或 `backend/app/nexus/data/`）下的若干条法规文本 + 一份 `README` 说明每篇覆盖的实体/边，便于回归核对。

### 任务 2 · 对测试文档建索引（依赖 A1 模型 + A2 管线）

- 按 A2 管线跑通：**切块（赋 fullname）→ 抽实体归一 → LLM 建边 → 向量化 → 落库**。
- 三个存储都要落数据：**实体索引**、**边表/图存储（SQL `nexus.edge`）**、**块向量库（AI Search）**。
- 提供一个可重复执行的索引入口脚本（如 `backend/app/nexus/index_cli.py` 或 `POST /index`），支持对测试集一键重建。
- 产出：测试集完成索引，可通过后端查询接口读回实体 / 边 / 块。

### 任务 3 · 前端「图浏览」组件（Vue Flow，强制拆分组件）

- 用 **Vue Flow** 可视化两层图：实体节点 + 关系边，点选可查看细节、下钻到原文块。
- **硬性要求：拆分成多个小组件**，避免一个巨型组件；按职责隔离，便于单独调整。建议结构：
  - `GraphExplorerPage.vue`：页面容器，负责布局与状态编排（不写具体渲染逻辑）；
  - `GraphCanvas.vue`：Vue Flow 画布封装（节点/边渲染、缩放、布局）；
  - `nodes/EntityNode.vue`：自定义实体节点（按 `AppType/Reg/Category` 区分样式）；
  - `edges/RelationEdge.vue`：自定义关系边（按 `requires/belongs_to/evidence…` 着色 + 权重）；
  - `EntityDetailPanel.vue`：选中实体的详情（别名、属性、status、关联边列表）；
  - `BlockViewer.vue`：原文块查看（text + fullname 溯源）；
  - `GraphToolbar.vue`：按实体类型 / 边类型过滤、跳数控制、搜索定位；
  - `IndexStatsPanel.vue`：索引概览（实体数 / 边数 / 块数）。
  - 数据层用 composable 隔离：`composables/useGraphData.ts`（拉取实体/边/块）+ 后端 API 封装，UI 组件不直连网络。
- 产出：登录后可进入「图浏览」页，看到测试集的实体关联、边类型、并能下钻到原文块。

**A3 里程碑**：挑好测试文档 → 一键建索引 → 前端图上看到实体、边、原文块，可交互下钻。

# 第二部分 · 查询索引阶段（在线检索）

> 目标：自然语言问题 → 可见的算子图(SQG) → 优化器 → 执行器，在两层图上执行，逐条溯源、可排查。

## B0 · 查询侧决策点（对应设计 §5）

- **逻辑算子子集**：一期只做 `检索 / 对比 / 回答`（覆盖设计中两个端到端示例）
- **优化器范围**：一期只做「校验 + 绑定 + 惰性 Ground + 并行」；谓词下推 / 公共子表达式复用二期

**产出**：`docs/decisions.md`（查询部分）。

---

## B1 · 物理算子（执行零件，对应 §3.5）

一期约 6 类：

- `Search`：一次 AI Search 调用（吞掉检索 / 过滤 / 重排 / 排序）——精确模式 & 向量模式
- `Lift` / `Ground`：块 ↔ 实体（沿 evidence 边）
- `Traverse(edge, dir, hops)`：图遍历（必须自建）
- 集合运算 `Intersect / Diff / Union / Dedup`（纯内存）
- LLM 加工 `LLM_Generate / Summarize / Compare`（固定模板）
- 校验 `SetCheck / ProvenanceCheck / Abstain`（可后置到二期）

每个算子统一签名（输入结果集 → 输出结果集 + 回填实际命中，供 §3.7 排查）。

**里程碑**：手写物理计划跑通设计例①的 `Search → Traverse → Diff → Ground → Answer`。

---

## B2 · SQG 编译器（LLM，对应 §3.1–3.3）

- 逻辑算子目录 + 算子 JSON schema（`op` / `desc` / `inputs` 三字段）
- Prompt + catalog：把实体类型、边类型、算子集喂给 LLM，让它直接输出 SQG
  （靠 catalog 让 LLM 出对结果，不在编译器里做重映射 / 硬编码纠偏）
- 输出校验：算子合法性、**必须无环**、输入齐全

**里程碑**：自然语言问题 → 合法 SQG JSON（对上设计中两个示例）。

---

## B3 · 优化器 + 执行器（对应 §3.4、§3.6）

- **逻辑算子 → 物理算子** 一对多映射展开
- 优化：惰性 Ground（中间只用 id，最终才取原文）、独立分支并行、结果缓存
- 执行器：拓扑序执行，回填每个算子实际输出（命中数 / fullname 列表）
- `校验` 不过时的有界回环重规划（一期可先不做回环）

**里程碑**：端到端跑通设计例①（精确入口 · 差集）与例②（向量入口 · 交集），中间数据与设计一致。

---

## B4 · 查询 API + 前端可视化（透明与溯源，对应 §3.7）

- 后端端点：`POST /query`，返回 SQG + 物理计划 + 逐算子回填 + 最终答案与溯源；
  挂到 `config.route_prefixes`
- 前端（Vue）：查询框 → **执行前展示 SQG 图**（Mermaid / 图）→ **执行后每算子回填**
  （命中数、fullname）→ 答案卡片带 fullname 出处链接
- .NET BFF 透传 + MSAL 认证

**里程碑**：网页里提问，看到「大模型想走的每一步 + 实际走的结果 + 带出处的答案」。

---

## B5 · 评估与打磨（对应 §5.6）

- 题集 + 指标：计划正确率 / 覆盖率 / 漂移率 / 溯源率
- 补齐 `supersedes / references` 边、`筛选 / 关联 / 校验` 算子、优化器谓词下推等二期项
