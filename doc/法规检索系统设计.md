# 法规检索系统设计 · Assertion-first 最终架构

> 状态：2026-07-13 重建后的实现依据。数据库细节以 [数据库表结构设计.md](数据库表结构设计.md) 为准。

## 1. 核心原则

1. **Assertion 是法规事实**：主体、行动、模态、条件、例外和适用范围必须保存在同一个原子事实中。
2. **Block 是证据**：每条 Assertion 通过精确 Quote 和位置关联一个 Primary Block，可另有 Supporting Block。
3. **Entity 和 Action 是稳定词汇**：Entity 表示法规、机构、活动、产品、分类和概念；Action 表示可跨主体比较的完整行动。
4. **Graph 是派生导航**：图边由 Accepted Assertion 派生，`graph_edge_support` 记录支持 Assertion；图边不是事实真值。
5. **Generation 原子发布**：每个 Store 只公开一个 Active Generation；构建中、失败和取消的代次不可查询。
6. **Collection 是查询安全边界**：一次查询只选一个 Collection，并冻结成员 Store 与各自 Active Generation。
7. **查询计划可见**：SQG 表达“查什么”，PEP 表达“怎么查”；协调器只执行，生成器不检索。

## 2. 索引模型

### 2.1 完整隔离代次

用户可以有一份上传一份。每次上传按 `category + 文件标题` 识别逻辑文档：新标题表示新增，同一标题表示替换；当前 Active Generation 中未上传的文档自动继承。系统内部仍为合并后的完整文档集合建立一个隔离 Generation：

```text
seed_candidate（继承当前 Active Generation 中未变更文档的事实与向量）
       ↓
parse
  ├─ embed（Block 写 AI Search，带 generation_id）
  └─ extract fanout（严格 Assertion JSON，最多一次反馈重试）
       ↓
resolve/persist（Entity、Action、Assertion、精确 Evidence）
       ↓
derive_graph（只从 Accepted Assertion 派生）
       ↓
quality_gate
       ↓
activate（原子切换 Store.active_generation_id）
       ↓
finalize
```

如果上传文件与当前同一逻辑文档的内容哈希完全一致，直接继承，不重复抽取和向量化。构建开始时冻结 `base_generation_id`；激活前再次比较 Store 指针，避免两个并发索引运行互相覆盖。删除文档和替换全部 Store 内容必须使用显式操作，不能由“未上传”隐式推断。

质量门禁至少检查：Block 清单与 AI Search 数量一致、抽取失败率、Assertion Primary Evidence、Participant 解析、Action/Entity 存在、Graph Edge Support 完整。

### 2.2 Assertion

Assertion 类型：`norm / definition / relation / deadline / penalty`。模态：
`must / must_not / may / should / factual / conditional_may`。

`norm` 至少有 Subject 和 Action；条件许可必须有 Condition。Evidence Quote 必须和 Block 原文指定 Span 一致。合法空抽取必须显式返回 `empty`，JSON/校验/服务失败不能伪装为空。

### 2.3 Action 集合比较

Action 保存完整规范化行动及 Participant，不包含模态、条件和例外。主体之间比较义务/许可/禁止时，以 `action_id` 为集合键，同时保留：

- 每个主体支持该 Action 的 Assertion IDs；
- 每条 Assertion 的模态、条件、例外和参与者；
- 集合运算类型、左右分支和结果数量。

因此交集、差集和并集不会丢失“哪个主体由哪条法规事实支持”。

## 3. Store、Collection 与查询快照

`search_store` 保存 AI Search 凭据、索引名和 `active_generation_id`。`collection_store` 维护 Collection 与 Store 多对多关系，`collection_access` 维护公开性以外的用户/角色可见性和用户默认项。

Initializer 在任何读取前冻结：

```json
{
  "collection_id": "clinical-regulations",
  "allowed_stores": ["store-a", "store-b"],
  "generation_scope": {
    "store-a": "gen-101",
    "store-b": "gen-208"
  }
}
```

规则：

- Collection 没有 Store，或任一 Store 没有 Active Generation，初始化失败；
- SQL Repository 的每个查询都从 `(store_id, generation_id)` CTE 开始；
- AI Search 每次调用必须显式传 Store 的冻结 Generation；
- 文档筛选先绑定冻结代次内的 `document/document_version`，再下推 `document_id`；
- PEP 只能缩小范围，不能更换 Collection、Store 或 Generation；
- `query_run` 保存 `allowed_stores`、`generation_scope` 和预算，历史运行保持可解释。

## 4. 查询五阶段

### 4.1 Initializer

确定一个当前用户可见 Collection。优先使用用户指定项，其次用户默认项、唯一可见项；多项且无默认时才用受限语义路由。随后冻结 Generation Scope，并读取该范围内可见的 Entity、Action、Document Version、Graph Relation 类型和各 Store 向量维度。

### 4.2 SQG Compiler

LLM 只输出一个有判别字段的强类型逻辑意图，不输出自由 `goal` 字典，不接触物理术语。

| 强类型意图 | 含义 |
|---|---|
| `find_subject_facts` | 查命名主体的 Assertion 或 Action，可带模态/类型/Predicate |
| `compare_subjects` | 比较两个以上命名主体的 Assertion/Action，做交/差/并 |
| `find_action_subjects` | 从命名 Action 反查 Subject |
| `traverse_relation` | 沿明确、可见的派生 Graph Relation 查相邻节点 |
| `compare_documents` | 比较两份以上明确命名文档，保留独立证据组 |
| `semantic_evidence` | 无法绑定结构事实时做开放语义证据检索 |

所有命名 Subject、Action、Document 和 Relation 必须唯一绑定可见词汇。首次校验失败时把反馈交给 LLM 重试一次；最终失败持久化原始无效输出并结算 Token。

### 4.3 PEP Optimizer / Planner

Planner 不调用 LLM。它先确定性绑定稳定 ID，再选择固定模板：

| SQG 意图 | 确定性 PEP 模板 |
|---|---|
| 单主体 Assertion | `EntityLookup → SubjectAssertions → [FilterModality] → GroundAssertions` |
| 单主体 Action | `EntityLookup → SubjectActions → [FilterModality] → GroundAssertions` |
| 多主体比较 | 每主体独立分支 → `Intersect/Diff/Union` 链 → `GroundAssertions` |
| Action 反查主体 | `ActionLookup → ActionSubjects → GroundAssertions` |
| 派生关系遍历 | `EntityLookup/ActionLookup → GraphTraverse → GroundAssertions` |
| 文档比较 | 每文档独立 `BlockSearch(document_id)` → `EvidenceBundle` |
| 开放证据 | `BlockSearch` |

PEP 节点显式声明输入端口的上游 Node ID 和数据类型，输出也声明类型。顶层必须显式绑定 `outputs.facts` 与 `outputs.evidence`。合法数据类型包括 `entity_set / action_set / fact_set / evidence_set / evidence_bundle`。

### 4.4 Workflow Coordinator

Coordinator 只把 PEP 翻译为通用 Workflow Node。通用 `services/workflow` 负责并行、依赖失败跳过、取消、节点 Token 与耗时；Coordinator 不重解释问题、不改计划、不生成答案。

### 4.5 Answer Generator

Generator 只收到 PEP 显式绑定的 Facts/Evidence，不持有 Repository 或 Search Adapter，因此无法检索。

- Assertion Citation：`assertion_id + block_key + quote`；
- 文档比较 Citation：`group + block_key + quote`；
- Citation 的 ID/Block 必须来自输入 Evidence；Quote 必须是输入 Quote/Text 的原文子串；
- 回答必须保留 Modality、Condition、Exception 和 Scope；
- 差集只表示在当前 Collection 冻结快照中未出现在右侧，不能扩展成普遍“不存在”。

## 5. 物理算子

| 算子 | 输入 → 输出 | 数据来源/行为 |
|---|---|---|
| `EntityLookup` | 无 → `entity_set` | 只绑定 Initializer 可见 Entity ID |
| `ActionLookup` | 无 → `action_set` | 只绑定 Initializer 可见 Action ID |
| `SubjectAssertions` | Entity → `fact_set` | 查询 Subject 的完整 Assertion |
| `SubjectActions` | Entity → `fact_set` | 以 Action ID 聚合，保留各 Subject 支持 Assertion |
| `ActionSubjects` | Action → `fact_set` | 反查支持该 Action 的 Subject |
| `AssertionSearch` | 无 → `fact_set` | 按强类型字段查询 Assertion |
| `GraphTraverse` | Entity/Action → `fact_set` | 只走明确 Relation，并保留支持 Assertion IDs |
| `FilterModality` | Facts → `fact_set` | 纯内存模态筛选 |
| `Intersect/Diff/Union` | Facts×Facts → `fact_set` | 按稳定 Comparison Key，合并 Provenance |
| `GroundAssertions` | Facts → `evidence_set` | 只取结果 Assertion 的 Primary/Supporting Quote 与 Block 元数据 |
| `BlockSearch` | 无 → `evidence_set` | 对每个 Store 按冻结 Generation 扇出，文档过滤不越代次 |
| `EvidenceBundle` | Evidence×N → `evidence_bundle` | 保留文档分支，不扁平丢失归属 |

`GroundAssertions` 不使用 Entity 的其他出处；它只返回结果事实自身的 Assertion Evidence，避免用无关语境佐证关系或集合结果。

## 6. 图 API

图节点可以是 `entity` 或 `action`，图边来自 `graph_edge`，支持信息来自 `graph_edge_support → legal_assertion → assertion_evidence`。

API 保持外层 `{nodes, edges}`，并提供 Catalog、Neighborhood、Node Detail、Edge Detail 和按 `block_key` 读取原文。未提供 Collection 时，只允许使用当前用户唯一可见或默认 Collection；绝不回退到全部 Generation。旧手工 Entity 创建和 Evidence Attach 不属于 Assertion-first 模型，接口返回明确 Unsupported。

## 7. 运行记录与失败语义

`query_run` 保存总体状态、快照、预算、Token、答案和 Citation；`query_stage` 固定保存五阶段；`query_node` 保存 PEP 物理节点。

- 当前阶段失败/取消后，后续 Stage 记为 `skipped`；
- Workflow 父节点失败时下游 Node 跳过；
- Compiler 最终失败保存 Raw Invalid Output；Generator 最终失败保存 Raw Output；
- LLM 返回后即使后续 JSON/契约校验失败，也必须 `pop_usage` 结算 Token；
- AI Search、SQL、凭据错误是失败，不得降级为空结果。

## 8. 模块边界

最终实现只有四个 Nexus 包：

- `nexus/domain`：Assertion-first 与 Store/Collection/QueryContext 契约；
- `nexus/infrastructure`：SQL、AI Search、Chat、Embedding、Store/Collection、Query、Graph Repository；
- `nexus/indexing`：完整 Generation 索引 Workflow；
- `nexus/querying`：五阶段查询、强类型 SQG、确定性 PEP、算子与记录器。

通用 `app/services/workflow` 保持独立且不包含 Nexus 业务语义。旧 `models / stores / index / query / llm / core` 实现已删除。
