# Nexus 数据库表结构设计

> 状态：索引与查询重建的数据库单一事实来源。数据库允许清空重建，不迁移旧 `entity_edge / evidence / index_run / query_run` 数据，也不使用 `v2` 命名。

## 1. 设计原则

1. **Block 是证据**：原文和向量保存在 Azure AI Search；SQL 保存 Block Manifest、稳定结构地址和写入状态。
2. **Assertion 是事实**：法规事实必须保存主体、行动、模态、条件、例外、原文 Quote 和位置。
3. **Entity 是稳定概念**：只保存可跨条文复用的法规、机构、活动、产品、分类和术语。
4. **Action 可做集合比较**：义务交集/差集比较 `action_id`，不比较“备案、报告”等脱离上下文的短语。
5. **Graph 是派生数据**：图边由有效 Assertion 派生，并通过支持表关联一个或多个 Assertion。
6. **Generation 原子发布**：构建中的数据不可查询；质量门禁通过后一次切换 Store 的 Active Generation。
7. **Collection 是安全边界**：一次查询只使用一个 Collection；运行时冻结 Store 和 Active Generation 快照。
8. **无数据库外键**：依赖完整性、删除顺序和原子激活由应用服务与质量门禁负责；保留主键、唯一索引和检查约束。
9. **运行过程可恢复**：索引保存 `run → node`，查询保存 `run → 五个 stage → physical node`。
10. **失败不可伪装为空**：每次 Block 抽取调用都保存 Attempt；非法 JSON、校验失败和合法空结果严格区分。

## 3. Active Generation 与查询范围

`search_store.active_generation_id` 是每个 Store 唯一的线上入口。

索引构建流程：

```text
building → validating → active
                    ↘ failed
                    ↘ cancelled
旧 active → retired
```

查询初始化时生成不可变快照：

```json
{
  "allowed_stores": ["store-a", "store-b"],
  "generation_scope": {
    "store-a": "generation-1",
    "store-b": "generation-7"
  }
}
```

所有 Assertion、Graph 和 Block 检索必须同时受 Store 与 Generation 限制。构建失败或取消时不修改 `active_generation_id`。

## 4. 文档与 Block

### 4.1 `document`

稳定逻辑身份，不随 Generation 改变。主要字段：

- `document_id`
- `canonical_title`
- `category`
- `source_uri`

### 4.2 `document_version`

同一文档在某个 Generation 中的快照：

- `document_version_id`
- `generation_id`
- `document_id`
- `content_hash`
- 标题、分类、来源快照
- `block_count`
- `state = staged | validated | rejected`

同一 Generation 中一个逻辑文档只能出现一次。

### 4.3 `block_manifest`

`block_key` 是 AI Search 文档键；`block_id` 是跨 Generation 稳定的法规结构地址。建议：

```text
文档:article-34
文档:article-34:paragraph-1
文档:article-34:paragraph-1:item-2
```

SQL 不保存完整 Block 原文，保存：

- Generation、Document Version、Document；
- 条、款、项、章节路径、父 Block；
- 原文 Hash、字符数、显示顺序；
- `extraction_state`；
- `search_state`。

## 5. 抽取审计

`block_extraction_attempt` 每调用一次 LLM 写一行，保留：

- Attempt 序号；
- Prompt/Extractor 版本；
- `succeeded | empty | invalid | failed`；
- 原始 JSON；
- 结构或语义校验错误；
- Token 与耗时。

合法空结果必须是显式 `empty`；异常返回 `{}` 不得记作成功。两次抽取后仍无法形成任何有效 Assertion 的 Block 记为 `quarantined`：保留完整审计但不进入事实索引。质量门禁允许最多 `max(1, 5% Block 数)` 个隔离块，超过阈值则拒绝发布，避免模型整体故障时产生残缺索引。

## 6. Entity 与归一

### 6.1 `entity`

允许类型：

- `Reg`
- `Org`
- `Activity`
- `Product`
- `Category`
- `Concept`

`lifecycle_state` 区分 `candidate / active / rejected`。构建中的新实体先为 Candidate；Generation 激活时，仅将被有效 Assertion 使用的实体提升为 Active。

### 6.2 `entity_alias`

别名带 `generation_id` 来源。失败 Generation 产生的 Alias 不参与线上查询或后续正常归一；人工 Alias 可以不绑定 Generation。

### 6.3 `entity_mention`

保存 Block 内局部提及、Span、候选、归一状态和最终 `entity_id`。归一状态：

- `pending`
- `matched`
- `new`
- `ambiguous`
- `rejected`

质量门禁不允许 Accepted Assertion 引用未解析 Participant。

## 7. Action

`action` 表示可比较的完整行动，例如：

- 向接种单位供应疫苗；
- 制定本行政区域免疫规划疫苗使用计划；
- 将使用计划报卫生健康主管部门备案。

它不保存 Assertion 特有的“应当、可以、不得、条件、例外”。这些属于 `legal_assertion`。

`action.signature_hash` 根据规范化 Verb 和 Participant 结构计算，作为全局去重键。

`action_participant` 支持：

- `object`
- `recipient`
- `authority`
- `beneficiary`
- `instrument`
- `target`

`action_mention` 保存每个 Block 中 Action 的原始抽取和归一过程。

## 8. Legal Assertion

`legal_assertion` 是事实主表。

### 8.1 Assertion Kind

- `norm`：义务、禁止、许可；
- `definition`：术语定义；
- `relation`：引用、归属、适用、替代；
- `deadline`：时限；
- `penalty`：法律责任。

### 8.2 Modality

- `must`
- `must_not`
- `may`
- `should`
- `factual`
- `conditional_may`

### 8.3 结构

Assertion 保存：

- `predicate`
- `action_id`
- `condition_text`
- `exception_text`
- `scope_text`
- 类型特有的 `payload` JSON
- `confidence`
- `accepted | review | rejected`

`assertion_entity` 保存 Participant：

- `subject`
- `object`
- `recipient`
- `authority`
- `beneficiary`
- `regulation`
- `activity`
- `product`
- `term`

`assertion_evidence` 保存一个或多个原文 Quote，并强制每条 Accepted Assertion 至少存在一条 `primary` Evidence。Quote 必须能在对应 Block 原文中定位。

## 9. 派生图

`graph_edge` 的两端可以是 `entity` 或 `action`。第一期边类型由领域目录控制，例如：

- `has_obligation`
- `has_prohibition`
- `has_permission`
- `applies_to`
- `belongs_to`
- `references`
- `supersedes`
- `performs_activity`
- `regulated_by`

`graph_edge_support` 把一条图边关联到一个或多个 Assertion。图边不保存 Evidence JSON，也不是最终事实来源。

## 10. 运行记录

### 10.1 索引

`index_run.dag` 保存 Workflow 结构，`index_node` 保存运行状态。运行记录增加：

- `generation_id`
- Entity、Action、Assertion、Graph Edge 数量；
- 质量问题数量；
- 当前 Phase。

`index_quality_metric` 保存硬门禁和警告，不另建独立测试系统。

### 10.2 查询

固定五阶段：

1. `initializer`
2. `compiler`
3. `optimizer`
4. `coordinator`
5. `generator`

`query_run` 保存 `allowed_stores` 和 `generation_scope` 快照。`query_stage` 保存强类型 SQG、PEP 及各阶段结果；`query_node` 保存 PEP 物理节点执行状态。

最终查询契约补充：

- SQG 是单个带 `kind` 判别字段的强类型 Intent，不保存自由 `goal` 字典；
- PEP Node 保存明确 Operator、类型化输入端口和输出类型，顶层显式绑定 `facts/evidence`；
- Planner 根据 Intent 使用固定模板确定性生成 PEP，不由 LLM 自由生成完整物理计划；
- `query_run.budgets` 保存实体、Block、Token 预算；
- Assertion Citation 为 `assertion_id + block_key + quote`，文档比较 Citation 为 `group + block_key + quote`；
- `GroundAssertions` 只读取结果 Assertion 的 `assertion_evidence`，不读取 Entity 的其他来源；
- Graph API 的每次查询也必须先解析一个可见 Collection 并冻结 Active Generation，不允许默认读取所有代次。

## 11. 必须由应用保证的不变量

由于数据库不使用外键，应用服务和质量门禁必须保证：

1. `search_store.active_generation_id` 指向该 Store 的 `active` Generation；
2. 同一 Store 同时最多一个 Active Generation；
3. Accepted Assertion 至少一个 Primary Evidence；
4. Assertion Evidence、Mention、Graph Edge 均属于同一 Generation；
5. Graph Edge 至少一个有效 Assertion Support；
6. Action 和 Entity Participant 必须存在；
7. AI Search 中的 Block 数和 `block_manifest.search_state='written'` 一致；
8. 查询只读取 Collection 内 Store 的 Active Generation；
9. Failed/Cancelled Generation 不得影响线上可见性；
10. Retired Generation 只有在不再被运行快照引用后才能清理。

## 12. 重建边界

建表脚本会删除并重建本设计中的所有表，不处理以下系统表：

- `nexus.api_log`
- `nexus.app_credential`
- `nexus.app_user`

旧索引、旧实体、旧边、旧出处、旧索引运行和旧查询运行全部丢弃。Store、Collection 结构保持兼容，但表内配置也会清空，重建后重新配置。
